Introduction & Learning Objectives

A compact, exam-focused introduction followed by interactive tools and quick activities to help you remember key concepts.

Purpose

Software is rarely written in one step; it requires planning, design, testing and maintenance. A Software Development Life Cycle (SDLC) gives developers a structured approach so the final system meets user requirements, is delivered on time and within budget, and is easier to maintain and enhance.

Different Life Cycles
Waterfall Model — key points
  • Principle: linear stages→ Requirements → Design → Implementation → Testing → Deployment → Maintenance.
  • Benefits: easy to manage; clear documentation.
  • Drawbacks: hard to go back; costly to change; poor for unclear requirements.
Iterative Development — key points
  • Principle: develop in small cycles (versions), test & refine each one.
  • Benefits: early user feedback; reduced project risk; easier feature addition.
  • Drawbacks: needs more user communication; can lengthen if too many iterations.
RAD (Rapid Application Development) — key points
  • Principle: build a prototype quickly and refine it with user feedback.
  • Benefits: fast; user involvement; flexible requirements.
  • Drawbacks: may be under-documented; not ideal for large safety-critical systems.
Program Development Cycle (stages)
  1. Analysis — gather requirements; identify inputs, processes, outputs.
  2. Design — plan solution using structure charts, pseudocode, flowcharts.
  3. Coding — write the program in the chosen language.
  4. Testing — verify behaviour with varied test data.
  5. Maintenance — fix, adapt or enhance after release.
Key Diagrams & Concepts
  • Structure charts — show program decomposition and parameters passed between modules (top-down).
  • State-transition diagrams — document states and how events change them.
  • Errors — syntax, logic, run-time; avoid with design, peer review, testing.
  • Testing methods — dry-run, walkthrough, white-box, black-box, integration, alpha, beta, acceptance, stub.
  • Test plan — what to test, test data, expected results, resources and schedule.
  • Maintenance — corrective, adaptive, perfective.

15.02 Program Design using Structure Charts

  

    A structure chart is a diagram that shows how a program is divided into 
    modules and how data (parameters) flows between them.  
    Each box represents a module or sub-task. Arrows show values passed up or down.
  


  

    
Key Terms

    
    
      
  • Structure chart: a graphical representation of the modular structure of a solution.
    • 
      
  • Parameter: a value passed between modules.
    • 
      
  • Interface: the communication between modules (using parameters).
    • 
    

  


  
  
Example 1: Average of Two Numbers

  

    Problem: Write a program that takes two numbers, calculates their average, and outputs the result.
  

  
   ┌───────────────┐
   │   Averages    │  (Main Module)
   └──────┬────────┘
          │
   ┌──────┴────────┐
   │   INPUT        │  (Read Number1, Number2)
   └──────┬────────┘
          │  Number1, Number2
   ┌──────┴────────┐
   │ Calculate Avg │  (Average = (N1 + N2) / 2)
   └──────┬────────┘
          │  Average
   ┌──────┴────────┐
   │   OUTPUT      │  (Display Average)
   └───────────────┘
  

  

    Explanation:  
    - Input module collects two numbers.  
    - Calculation module computes the average.  
    - Output module displays the result.  
    This chart demonstrates the flow of parameters from input to calculation, then back to output.
  

  

    Exam Tip: Structure charts often test your ability to show data flow using arrows.
  


  
  
Example 2: Number Guessing Game (Single Guess)

  

    Problem: User makes a single guess of a secret number. The program outputs either 
    "Congratulations" or "Try again".
  

  
   ┌────────────────────┐
   │  Number Guessing   │
   └─────────┬──────────┘
             │
   ┌─────────┴──────────┐
   │ Generate SecretNum │
   └─────────┬──────────┘
             │ SecretNum
   ┌─────────┴──────────┐
   │     INPUT Guess    │
   └─────────┬──────────┘
             │ Guess
   ┌─────────┴──────────┐
   │ Compare Guess = ?  │──True──► OUTPUT "Congratulations!"
   └─────────┬──────────┘
             │ False
             ▼
        OUTPUT "Try Again"
  

  

    Explanation:  
    - Secret number generated first.  
    - User input guess compared to secret number.  
    - Depending on True/False branch, output differs.  
  

  

    Exam Tip: Diamonds (◊) in structure charts represent decisions (True/False).
  


  
  
Example 3: Pyramid Drawing

  

    Problem: Print a pyramid of symbols with increasing rows until the maximum is reached.
  

  
   ┌───────────────┐
   │    Pyramid    │
   └──────┬────────┘
          │
   ┌──────┴────────┐
   │ Input MaxRows │
   └──────┬────────┘
          │ MaxRows
   ┌──────┴────────┐
   │ Initialise    │ (Row=1)
   └──────┬────────┘
          │
   ┌──────┴────────┐
   │ Loop UNTIL    │──► Repeats until Row=MaxRows
   └──────┬────────┘
          │
   ┌──────┴────────┐
   │ Output Spaces │
   └──────┬────────┘
          │
   ┌──────┴────────┐
   │ Output Symbols│
   └──────┬────────┘
          │
   ┌──────┴────────┐
   │ Adjust Row    │ (Row++)
   └───────────────┘
  

  

    Explanation:  
    - User inputs maximum number of rows.  
    - Each iteration outputs spaces then symbols.  
    - Repeats until pyramid is complete.  
  

  

    Exam Tip: Semi-circular arrows in structure charts indicate loops/repetition.
  


  
  
Example 4: Connect 4 Game

  

    Problem: Create a program for the Connect 4 game. Players take turns placing counters into columns until a winner is found.
  

  
   ┌────────────────────┐
   │     Connect 4      │
   └─────────┬──────────┘
             │
   ┌─────────┴──────────┐
   │ Initialise Board   │
   └─────────┬──────────┘
             │
   ┌─────────┴──────────┐
   │   Setup Game       │
   └─────────┬──────────┘
             │
   ┌─────────┴──────────┐
   │ WHILE Not Finished │
   └─────────┬──────────┘
             │
   ┌─────────┴──────────┐
   │ Player Move        │ (Choose Column)
   └───────┬────────────┘
           │
   ┌───────┴────────────┐
   │ Validate Column    │ (Check if valid)
   └───────┬────────────┘
           │
   ┌───────┴────────────┐
   │ Find Free Row      │
   └───────┬────────────┘
           │
   ┌───────┴────────────┐
   │ Update Board       │
   └───────┬────────────┘
           │
   ┌───────┴────────────┐
   │ Check Winner?      │──► Yes → OUTPUT "Winner Found"
   └────────────────────┘
  

  

    Explanation:  
    - The board is initialised and game setup occurs.  
    - Each turn: player chooses column → program checks validity → finds free row → places counter.  
    - Loop continues until game is finished (winner or draw).  
  

  

    Exam Tip: For larger programs (like games), structure charts are essential for 
    visualising module interaction and parameter passing.
