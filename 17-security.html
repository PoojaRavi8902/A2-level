Phase 1 — Starter (12–15 minutes)

Purpose: activate prior knowledge (Data representation / Networking). Quick formative check.
  1. Warm-up quiz (10 mins) — 6 quick MCQs about previous chapters (number systems, basic networking, basic crypto vocabulary). Students do individually on paper or use the self-marking quiz below.
  2. Peer check (2–3 mins) — students swap answers with a neighbour and discuss any differences.
  3. Teacher quick review (2 mins) — call 3 answers and note misconceptions on the board.
Teacher notes:
  • Use the quick quiz to identify who needs help with binary/hex conversions and basic networking terms (IP, TCP/UDP).
  • If many students miss a question, plan to briefly revise that point in the first lecture burst.

Phase 2 — Teacher Lecture 1 (10 minutes)

Focus:

Clear explanation of core security concepts students must know for the exam:

  • Symmetric vs Asymmetric encryption — definitions, when each is used, advantages/disadvantages.
  • Public Key Infrastructure (PKI) — simple certificate idea and the role of Certificate Authorities (CA).
  • SSL/TLS handshake (high-level) — purpose and the steps (client hello, server hello, cert, key exchange, secure session).
What to say (scripted)
  1. "Encryption transforms readable data into ciphertext so that only authorised parties can read it."
  2. "Symmetric encryption uses one shared key — it's fast, suitable for encrypting large streams (e.g. AES). But sharing the key securely is hard."
  3. "Asymmetric uses a public-private key pair. The public key encrypts, private key decrypts — it solves the key distribution problem at the cost of speed (e.g. RSA)."
  4. "Certificates are like digital ID cards issued by trusted authorities — they prove a website is who it claims to be."
  5. "TLS handshake: client and server agree protocols and keys, server presents certificate, session keys are derived and used for encryption."

Visual aids (draw on board or project):

  1. Diagram: Symmetric vs Asymmetric flows (label keys clearly).
  2. Simple TLS sequence diagram: ClientHello → ServerHello, Certificate → Key exchange → Encrypted messages.

Phase 3 — Student Activity (15 minutes)

Task A: Practical encryption exercise (paired work)

Students work in pairs. Give each pair a short plaintext message and a Caesar shift (teacher chooses). They must encrypt and then show the class the ciphertext and the shift used.

  1. Teacher prints or displays 6 different short phrases (10–12 characters).
  2. Each pair gets one phrase and a shift between 1–10.
  3. Pairs exchange ciphertexts with another pair and attempt to decrypt (testing symmetric understanding).

Task B: PKI Roleplay (optional extension - if time allows)

Choose 6 volunteers: CA, Website, Client, Attacker, Browser, Auditor. Roleplay the issuance and verification of a certificate and demonstrate what happens if the certificate is invalid.

Marking guidance (informal)
  • Correct ciphertext & decryption (3 marks)
  • Clear explanation of which key used (2 marks)
  • Participation & peer feedback (1 mark)

Phase 4 — Teacher Lecture 2 (12 minutes)

Focus:

  • Digital signatures — how they work, simple diagram (hash + private key = signature)
  • SSL/TLS in practice — how browsers show trust (chains of trust, revoked certs)
  • Intro to quantum cryptography — basic idea only and its implications for key distribution
Scripted explanation (what to say)
  1. "A digital signature signs a digest (hash) of the message with a private key — anyone with the public key can verify the signature matches the message."
  2. "Chain of trust: Root CA → Intermediate CA → End-entity cert. Browsers trust root CAs built into them."
  3. "Quantum key distribution uses quantum states; eavesdropping changes the state and is detectable — promising but costly."

Instructor demo (2–3 mins):

Open a browser to any HTTPS site, click the padlock and show certificate details. Point out issuer, validity period and subject.

Worked example (on board):

// Digital signature (conceptual)
message = "Pay 200 to A"
hash = H(message)
signature = encrypt_with_private_key(hash)
// Receiver: hash2 = H(message)
// verify = decrypt_with_public_key(signature)
// if hash2 == verify -> authentic

  3. Homework: Short written task — explain in 150 words how TLS protects online banking.
