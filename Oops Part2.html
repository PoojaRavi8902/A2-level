27.01 Concept of Object‑Oriented Programming (OOP)

In earlier chapters you learned procedural programming: code organised into procedures/functions, with data stored separately in variables or records. While effective for small programs, procedural systems scale poorly as complexity grows. Object‑Oriented Programming (OOP) addresses this by bundling data with the code that operates on it.

An object is a self‑contained unit that represents a real‑world entity or conceptual thing. Each object stores data called attributes and exposes behaviour via methods. The blueprint for creating objects is a class.

Encapsulation is the core idea: attributes are kept private, and access is controlled through public methods. This prevents accidental or invalid modification of internal state.

From Records to Objects

In procedural code, records group related fields, but any part of the program can directly modify them. OOP removes this direct access. Instead, you ask the object to perform an action.

# Procedural-style record access (conceptual)
car.engine_size = 2500  # direct and unsafe

# OOP-style
car.set_engine_size(2500)  # validated and controlled

Why OOP is Preferred

  • Improved reliability through controlled access
  • Reusability via classes and inheritance
  • Maintainability and clearer program structure
  • Alignment with real‑world modelling

27.02 Designing Classes and Objects

Design precedes coding. In exam questions, marks are awarded for clear class design before any syntax. Think in terms of nouns (classes) and verbs (methods).

Attributes, Getters, Setters

Every attribute that must be read needs a getter. Attributes that may change need a setter. Some attributes are fixed at creation time and should not have setters.

Constructor

The constructor (__init__) runs when an object is instantiated. It sets initial values and ensures the object starts life in a valid state.

Properties (Python)

Python properties provide a clean interface while preserving encapsulation. Examiners like this approach.

class Example:
    def __init__(self, x):
        self.__x = x

    @property
    def x(self):
        return self.__x
Exam Tip: Always state that a class is a user‑defined data type. This earns reflection and theory marks.

27.03 Writing Object‑Oriented Code in Python

This section focuses exclusively on Python 3, exactly as expected in Cambridge examinations.

The Car Class – Complete, Clean, Exam‑Ready

class Car:
    def __init__(self, vehicle_id, engine_size):
        self.__vehicle_id = vehicle_id
        self.__engine_size = engine_size
        self.__registration = ""
        self.__date_of_registration = None
        self.__purchase_price = 0.0

    @property
    def vehicle_id(self):
        return self.__vehicle_id

    @property
    def engine_size(self):
        return self.__engine_size

    @property
    def registration(self):
        return self.__registration

    @registration.setter
    def registration(self, value):
        self.__registration = value

    @property
    def purchase_price(self):
        return self.__purchase_price

    @purchase_price.setter
    def purchase_price(self, value):
        self.__purchase_price = value
Common Exam Error: Forgetting self as the first parameter in every method.

27.04 Inheritance

Inheritance allows new classes to be created from existing ones. The original class is the base class; the new class is the subclass.

Python Implementation

import datetime

class LibraryItem:
    def __init__(self, title, author_artist, item_id):
        self.__title = title
        self.__author_artist = author_artist
        self.__item_id = item_id
        self.__on_loan = False
        self.__due_date = None

    def borrowing(self):
        self.__on_loan = True
        self.__due_date = datetime.date.today() + datetime.timedelta(weeks=3)

    def returning(self):
        self.__on_loan = False

27.05 Polymorphism

Polymorphism means one method name, different behaviour depending on the object’s class.

class Book(LibraryItem):
    def __init__(self, title, author, item_id):
        super().__init__(title, author, item_id)
        self.__is_requested = False

    def print_details(self):
        print("Book Details")
        super().print_details()
        print(self.__is_requested)
Exam Phrase to Memorise: “Polymorphism allows a method to behave differently for different classes in the same inheritance hierarchy.”
